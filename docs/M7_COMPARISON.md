# M7 — 3단계 종합 비교: 각 단계의 "의도"와 "진행"

> 이 프로젝트는 **같은 개활 창고·같은 시나리오에서 localization 방식만 바꿔** 문제→완화→해결을 보인다.
> 이 문서는 각 단계가 **무엇을 의도했고, 어떻게 작동하며, 실제로 어떻게 진행됐는지**를 한 곳에 정리한다.

## 1. 각 단계의 의도·작동·결과

### 0단계 · Baseline (AMCL only) — **"문제를 눈으로 보인다"**
- **의도**: feature 가 없거나 반복되는 개활 창고에서 표준 실내 로컬라이제이션(AMCL)이 **실패함을 정량 확립**.
  이게 프로젝트의 출발점(문제 제기).
- **작동**: 파티클 필터가 사전 지도 ↔ 2D LiDAR 스캔을 정합해 위치 보정. 개활 core 에선 12m 안에 아무 특징이
  없어(유효빔 0) **보정 근거가 사라지고**, 20m 격자의 **똑같은 기둥**을 봐도 엉뚱한 기둥에 매칭(perceptual aliasing).
- **결과**: 개활 횡단 ATE **24~34m**, 공분산 폭발(9~25m), 한 번 잃으면 회복 불가. → **절대 위치 실패.**

### 1단계 · +Noise (노이즈만, N) — **"드리프트를 현실화한다"(EKF 의 전제)**
- **의도**: 시뮬 오도메트리는 슬립 0 이라 **너무 완벽** → 그대로면 EKF 가 완화할 대상이 없다. 그래서 바퀴 슬립·
  **헤딩 바이어스**를 인공 주입해 현실적 추측항법 드리프트를 만든다. N 은 그 자체가 목적이 아니라 **EKF 효과를
  재려는 control(대조군)**: "노이즈만 있을 때"와 "노이즈+EKF"를 같은 seed 로 비교하려는 것.
- **작동**: `noise_injector` 가 깨끗한 속도에 스케일·랜덤 + **회전 상수 바이어스 `b_ω`**(헤딩이 서서히 틀어지는
  킬러)를 실어 재적분, 이를 **odom→base TF 에** 실는다(AMCL 모션모델이 읽는 곳).
- **결과**: aliasing(기존) 위에 추측항법 드리프트가 **중첩**. ATE 는 여전히 큼(단일 런이라 aliasing 난수로 편차).

### 2단계 · +EKF (노이즈+EKF, E) — **"융합으로 드리프트를 늦춘다(완화)"**
- **의도**: 오도메트리+IMU 를 `robot_localization` EKF 로 융합해 **추측항법 드리프트를 얼마나 완화하나**를 정량화.
  근본 해결이 아니라 "보정 없이 버티는 시간 늘리기".
- **작동**: 바퀴 오도의 **속도**와 IMU 의 **절대 yaw**(정확)를 융합 → **헤딩을 앵커링**해 `b_ω` 를 상쇄, odom→base
  추측항법을 개선. AMCL 은 그대로(map→odom 담당).
- **결과(혼합)**: **드리프트 지배 국면(C4)은 완전 복구**(ATE 10.5→2.3m, drift→0). 그러나 **개활 횡단(C1·C3·C5)은
  aliasing 대재앙이 지배**해 효과가 묻히고, **cov 는 여전히 큼** = 절대 위치는 못 고침. → **완화 ≠ 해결.**

### 3단계 · +Landmark (반사판, 해결) — **"절대 위치를 확정해 aliasing 을 없앤다"**
- **의도**: 개활지의 근본 문제 = **절대 위치의 부재**. 알려진 좌표의 인공 특징(반사판)을 심어, 3개 이상 보이면
  **삼각측량으로 절대 위치를 직접 확정** → aliasing 을 원천 제거. (실제 산업 AGV 방식.)
- **작동**: `landmark_localizer` 가 /scan 에서 **가는 폴(반사판)을 기하로 검출**(폭으로 기둥과 구별) →
  보이는 반사판들의 **상대 성좌를 landmarks.yaml 에 매칭**(pose 불변이라 aliasing 에도 견고) →
  ≥3 매칭이면 **2D Procrustes 최소자승**으로 절대 pose 확정 → **map→odom 보정**. EKF 는 보정 사이 추측항법으로 유지.
- **결과**: 개활 전 구간 ATE **0.13~0.32m**, cov 붕괴(~5cm), aliasing 점프 소멸. → **해결.**

## 2. 진행이 어떻게 이어지나 (한 줄 논리)

```
Baseline: 특징이 없다 → 스캔매칭이 절대위치를 못 잡는다 (문제)
   │  "오도로 버티면 되지 않나?" → 근데 시뮬 오도가 너무 완벽하니
  +Noise: 현실적 드리프트를 주입 (EKF 를 재기 위한 무대)
   │  "융합하면 드리프트를 늦출 수 있다"
  +EKF:  IMU 로 헤딩 드리프트는 잡음 → 그러나 aliasing(절대위치)은 그대로 (완화뿐)
   │  "상대 보정으론 부족하다 → 절대 관측이 필요하다"
  +Landmark: 알려진 반사판으로 절대위치 확정 → aliasing 제거 (해결)
```
각 단계는 **직전 단계의 한계를 정확히 겨냥**한다: 노이즈는 오도의 완벽함을, EKF 는 드리프트를, 랜드마크는
aliasing(절대위치)을. **cov(확신도)가 세 방식 중 랜드마크에서만 붕괴**하는 것이 "진짜 해결"의 지표다.

## 4. 종합 지표 & 결론

![3단계 종합: ATE(로그축) + cov](../ros2_ws/outputs/landmark/final_compare_bars.png)

| 단계 | 방법 | 개활 ATE | cov(확신도) | 판정 |
|------|------|---------|------------|------|
| Baseline | AMCL | 24~34m | 폭발 | **문제** |
| +EKF | +오도/IMU 융합 | 여전히 큼 | 여전히 큼 | **완화** |
| +Landmark | +반사판 삼각측량 | **0.13~0.32m** | **붕괴(~5cm)** | **해결** |

**결론**: 개활 창고처럼 feature 가 없거나 반복되면 **스캔매칭·센서융합만으로는 절대 위치를 못 잡는다.**
알려진 인공 특징(반사판)의 **절대 관측**이 있어야 localization 이 성립한다 — 이 프로젝트가 처음 던진 질문에 대한
정량적 답이자, 실제 산업용 AGV 가 반사판을 쓰는 이유의 재현.

> 한계(공통): 단일 런이라 baseline/EKF 의 ATE 는 aliasing 난수로 편차가 큼(엄밀히는 반복 평균 ⑥ 필요).
> 견고한 결론은 **경향**(문제→완화→해결)과 **cov 거동**(랜드마크만 붕괴)이다.
